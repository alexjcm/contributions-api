# Auth0 setup (API Hono)

## Tenant/App Details
- Domain: `dcm-platform.us.auth0.com`
- API Identifier (Audience): `https://api.dcm`
- Issuer: `https://dcm-platform.us.auth0.com/`

## Versioned tenant assets
- The manually managed/exported Auth0 tenant artifacts live in `dcm-api/auth0-tenant-config/`.
- That directory currently versions the tenant-facing `actions`, `emails`, and `roles` used as operational reference for Dashboard updates or future Deploy CLI workflows.

## Minimum Configuration in Auth0 Dashboard
1. Under `Applications > APIs > DCM Api > Settings`:
   - Enable RBAC: `ON`
   - Add Permissions in the Access Token: `ON`
   - Allow Offline Access: `ON` (Required for Refresh Tokens)
   - Custom Error Page (optional): Dashboard → Settings → Tenant Settings → Error Pages.

2. Create API Permissions:
   - `summary:read`
   - `contributions:read`
   - `contributions:write`
   - `contributors:read`
   - `contributors:write`
   - `settings:read`
   - `settings:write`
   - `auth0_sync:write`

3. Create Roles (`User Management > Roles`) and assign permissions:
   - `viewer`: `summary:read`, `contributions:read`, `contributors:read`
   - `admin`: `viewer` permissions + `contributions:write`, `contributors:write`, `settings:read`, `settings:write`
   - `superadmin`: Same permissions as `admin`, plus `auth0_sync:write`

4. Assign roles to users.

5. Account Linking:
   - **Logic:** Suggested Google ↔ canonical DCM account linking via Auth0 Actions and DCM API. The Action only suggests linking when it finds exactly one `dcm_managed` canonical account.
   - **Versioned Action Name:** `dcm-access-and-account-linking`
   - **Action Configuration:** Requires `SESSION_TOKEN_SECRET` (generated by the developer), `DCM_PWA_URL` (environment specific), `AUTH0_DOMAIN`, `M2M_CLIENT_ID`, and `M2M_CLIENT_SECRET`.
   - **API:** `POST /api/auth/link-token` endpoint for secure identity validation.
   - **Post-change-password Action Name:** `dcm-clear-password-setup-pending`
   - **Post-change-password Action Configuration:** Bind the versioned `dcm-clear-password-setup-pending` Action with `AUTH0_DOMAIN`, `M2M_CLIENT_ID`, and `M2M_CLIENT_SECRET` so the temporary onboarding flag is removed immediately after the user defines their first password.

6. Contributor lifecycle sync:
   - `AUTH0_VIEWER_ROLE_ID` must be configured in the Worker runtime.
   - `AUTH0_SPA_CLIENT_ID` must be configured in the Worker runtime and must point to the production SPA Auth0 application used for interactive login.
   - DCM marks canonical accounts with `app_metadata.dcm_managed = true`.
   - Newly created DB users also receive `app_metadata.dcm_password_setup_pending = true` until the first successful password definition removes it.
   - Deactivation does not delete users in Auth0; it leaves only the `viewer` role when auto-sync is enabled.
   - There is no manual contributor sync endpoint when auto-sync is disabled.

## Contributor lifecycle behavior

### Resolution order
- If DCM already has `auth0_user_id`, the backend resolves that user first.
- If it no longer resolves, the backend falls back to exact email lookup in Auth0.
- If multiple users share the same email and there is no single canonical `dcm_managed` account, the backend does not auto-pick one; it stores `error`.

### Auto-sync ON
- Existing valid social account or linked primary account: reuse it, ensure `viewer`, persist canonical `user_id`.
- Existing DB account with password already established: ensure `viewer`, status `linked`.
- Existing DB account without password reset yet: ensure `viewer`, status `pending_password`.
- No Auth0 account: create DB user, mark `dcm_managed`, mark `dcm_password_setup_pending`, assign `viewer`, trigger password reset, status `pending_password`.
- Deactivation: keep the Auth0 user, but leave only the `viewer` role.
- Special deactivation case: if the user already has zero roles in Auth0 because of an explicit administrative decision, DCM respects that state and persists `no_access` instead of re-adding `viewer`.

### Auto-sync OFF
- DCM persists the contributor locally only.
- No automatic calls to Auth0 are made.
- No manual sync path is available as a bypass.

### Change of email
- DCM no longer deletes Auth0 users because of email changes.
- DB users may have their email updated through the Management API in this phase.
- Social-primary users are not mutated automatically in Auth0 in this phase; DCM updates its own record and leaves Auth0 alignment as an operational follow-up if needed.

### Social-first guardrail
- If a user already exists only as a valid social identity and that identity will be used by DCM, the backend must provision that same account.
- DCM does not create a parallel DB account only to send a password reset email.

### Change-password template UX
- Auth0 only provides one Change Password template, so DCM uses Liquid conditionals in that template.
- If `user.app_metadata.dcm_password_setup_pending == true`, the template must render an account activation message instead of a normal password reset message.
- DCM clears `dcm_password_setup_pending` in a `post-change-password` Action as soon as the user defines their password. The post-login Action keeps a defensive fallback if the flag is still present and `last_password_reset` already exists.
- The reset email itself is triggered with `AUTH0_SPA_CLIENT_ID`, not the M2M client ID, so Universal Login can route the user back to the production SPA after a successful password change.
- The versioned body lives in `auth0-tenant-config/emails/reset_email.html`.
- The versioned subject lives in `auth0-tenant-config/emails/reset_email_subject.txt`.

## Implemented Authorization Behavior
- Middleware validates JWT (`iss`, `aud`, signature, expiration).
- Permission Extraction:
  - Primary: `permissions` claim
  - Fallback: `scope` claim
- Permission constants are defined in `src/config/permissions.ts`.
- `PUT /api/settings` is authorized by setting key:
  - `monthly_amount_cents` requires `settings:write`
  - `auth0_auto_sync_enabled` requires `auth0_sync:write`
